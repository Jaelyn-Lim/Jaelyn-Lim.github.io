URL 中自定义 DRF 的URL映射

在使用的时候，对于DRF自己本身的URL的设计会有些杂乱，有时候会直接在 Django 层面上进行URL的设计，这时候如果又继续想要用 DRF 的一些特性的时候，就需要在URL进行映射。

原本使用 DRF 的 router 的话就可以直接使用router.register(r'order', OrderListViewSet)就搞定了，不过如果要在 Django 的层面上使用的时候，就需要分别把 get 请求映射到list和retrieve上，特别的，在使用类似于retrieve等需要特指某个具体对象的时候，就需要指明参数，表明是依照这个参数寻找数据库的值的，默认是按照数据库的主键进行查询的。如果需要按照别的字段进行查询，需要使用lookup_field来定义，所定义的字段可以不在Serializer中，不过必须是数据库中的某列字段。

在使用 URL 的时候，可以发现，r'^order/(?P<pk>\d+)/$'，这样的表达式表示在 Django 匹配的时候，如果匹配到形如order/12/这样的URL的时候，就会把捕获到的值放到名为pk的关键字参数中。
但是，在试图中，就需要在参数列表中定义一样名字的参数，def order(request, pk):

总结一下，在URL中使用参数并且采用将获得的值赋值到一个参数变量这一方法的时候（推荐这种赋值的做法，当我们的URL参数很多的时候，我们就能很好的区别各个不同的参数，避免调用错误，也方便阅读），同时，在view视图中，需要我们在参数列表中定义相同名字的变量。如果要使用 DRF 的功能，就需要在ViewSet中定义lookup_field，默认是pk，表示数据库的主键，同样也可以改为别的值（不过最好就是使用能表示唯一性的主键pk）。

pipreqs

创建环境依赖包的列表文件

此工具是基于 imports，这是什么意思呢，即你的项目引入了哪个包，此工具才会把引入的包写到 requirements.txt 中，是不是觉得要比 pip freeze 干净，注意生成的是 requirements.txt 文件，而不是 requirement.txt
