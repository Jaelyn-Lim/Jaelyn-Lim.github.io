Docker是一个工具，可以更容易为任何应用封装一个分配的构件，部署量产化到任何环境当中，简化敏捷开发组织的工作流及响应。

为什么要用Docker：

  • 更高效的利用系统资源
  • 更快速的启动时间
  • 一致的运行环境
  • 持续交付和部署
  • 更轻松的迁移
  • 更轻松的维护和扩展

快速入门

镜像 是一种轻量级、可执行的独立软件包，它包含运行某个软件所需的所有内容，包括代码、运行时、库、环境变量和配置文件。

容器 是镜像的运行时实例 - 实际执行时镜像会在内存中变成什么。默认情况下，它完全独立于主机环境运行，仅在配置为访问主机文件和端口的情况下才执行此操作。

容器在主机内核上以本机方式运行应用。与仅通过管理程序对主机资源进行虚拟访问的虚拟机相比，它们具有更好的性能特征。容器可以获取本机访问，每个容器都在独立进程中运行，占用的内存不超过任何其他可执行文件。

镜像（Image）和容器（Container）的关系，就像是面向对象程序设计中的 类 和 实例 一样，镜像是静态的定义，容器是镜像运行时的实体。容器可以被创建、启动、停止、删除、暂停等。

这里对基本的命令总结一下：

  • FROM

所有Dockerfile都必须以 FROM 命令开始，这个命令会指定镜像基于哪个基础镜像创建，接下来的命令也会基于这个基础镜像，也可以使用多次，但是会创建多个镜像。

  • MAINTAINER

表示这个镜像的作者或者说维护者。

  • RUN

执行命令，是在shell或者exec的环境下执行的命令。

  • ADD 和 COPY

复制文件，两个命令都是在镜像中添加文件，不同的点是，ADD 可以将远程的文件添加到镜像中，会更加强大一点，两者都可以将本地的文件添加到镜像中。

  • ENTRYPOINT 和 CMD

CMD 设置容器启动后默认执行的命令及其参数，但 CMD 能够被 docker run 后面跟的命令行参数替换。ENTRYPOINT 配置容器启动时运行的命令。一般可以认为，一些不需要改变的参数可以用 ENTRYPOINT ，会改变的参数使用 CMD。

配置给容器一个可执行的命令，这意味着在每次使用镜像创建容器时一个特定的应用程序可以被设置为默认程序。同时也意味着该镜像每次被调用时仅能运行指定的应用。类似于CMD，Docker只允许一个ENTRYPOINT，多个ENTRYPOINT会抵消之前所有的指令，只执行最后的ENTRYPOINT指令。

容器

以前和现在

以前的APP部署和现在的部署方式区别。

以前的容器部署

  • 部署慢
  • 成本高
  • 资源浪费
  • 难于迁移扩展
  • 硬件限制

虚拟化优势

  • 资源池，一个物理机的资源分配到不同的虚拟机里
  • 容易扩展，加物理机器或加虚拟机
  • 容易云化，各种云服务器

局限性：每个虚拟机都是完整的操作系统，虚拟机增多的时候，会消耗过多的系统本身的资源。

容器优势

  • 解决开发和运维之间的矛盾

  • 在开发和运维之间搭建一个桥梁，是实现devops的最佳解决方案

  • 容器是APP层面的隔离，虚拟化是物理资源层面的隔离。
  • 对软件和起依赖进行标准化打包
  • 应用之间相互隔离
  • 共享一个 OS Kernel
  • 可以运行在多个主流的操作系统上

docker

Docker 是用 Go 语言实现的，属于操作系统层面上的虚拟技术。主要还是采用 Linux 下的 Namespace 资源隔离技术，是一种内核级别的环境隔离技术。

与传统的虚拟技术不同，传统的虚拟技术是虚拟出一套完整的硬件支持，在其基础上运行一个完整的操作系统，在这之后再把所需运行的程序运行。而容器内的应用进程直接运行于宿主的内核，容器内没有自己的内核，而且也没有进行硬件虚拟。因此容器要比传统虚拟机更为轻便。

Docker 镜像和容器

docker 架构和底层技术简介

  • docker 提供了一个开发，打包，运行app的平台
  • 把app和底层的infrastructure隔离开来

Docker Engine

  • 后台进程（dockerd）
  • REST API Server
  • CLI接口（docker）

Docker Architecture

底层技术支持

  • Namespaces：做隔离pid，net，ipc，mnt，uts
  • Control groups：做资源限制
  • Union file systems：Container 和 image 的分层

Docker Image

什么是 Image

  • 文件和 meta data 的集合（root filesystem）
  • 分层的，并且每一层都可以添加、改变、删除文件，成为一个新的 image
  • 不同的 image 可以共享相同的 layer
  • Image 本身是 read-only 的

在最底层的 Linux 上，创建多个分层的 Image ，之后的每一次改变都会创建一个新的 Image，在节省资源的同时，并且将不同的内容区分开来。

Image 获取的方式

  • 通过 Dockerfile 去 build 一个image。
  • 从 Registry 中拉取下来（pull from registry）sudo docker pull ubuntu:14.04

Docker hub 是一个类似于 GitHub 的一个平台，允许用户上传自己创建的 docker 。其中分两类，一个是官方提供的镜像，还有第三方。如果使用官方所提供的，可以直接使用 sudo docker pull ubuntu:14.04 镜像名称和版本号来下载（不写版本号默认最新），或者使用 用户名 / 内容 的方式来下载（ docker pull bitnami/workpreeeeess ）。

Container

是什么

  • 通过 Image 创建 （copy）
  • 在 Image layer 之上建立一个 container layer （可读写）
  • 类比面向对象：类和实例
  • Image 负责 app 的存储和分发，Container 负责运行 app

首先，Container 是基于 Image 来创建的，类似与类和实例的关系，先有类才有实例；再者，Image 是只允许读的，没有写的权限，所以要有一个 Container 来负责写；最后，在功能上来说，Image 负责 app 存储分发，Container 负责运行。

实践

例如，在 Centos 下安装 vim ，形成一个新的 container。

docker run -it centos 进入交互状态，执行安装指令 yum install -y vim ，退出。

之后需要做的是将基于这个已经安装好 vim 的 centos 变成一个 image。

使用命令 docker commit nostalgic_hawking lim/centos-vim 接收两个参数，一个是 container 的名称，另外一个是生成 image 的名称。（不写 tag 默认 latest）

不过通过对原有 container 修改所创建的 image 是不提倡的，因为这样对于使用者来说是透明的，并不知道这个 image 是怎么创建的，所以更好的方式是通过 docker file 去创建 image。

首先会基于原有的 base image ，在这之上在创建新的层。

由此可以看出，之所以能在 image 上执行 写（run yum install -y vim） 的操作，是因为在base image 上创建了一个临时的container，这个layer用来执行命令，在这个临时的layer里面去安装 vim 。

在执行完之后就会把这个临时的层删除，之后再基于那个临时的container，去commit一个新的image。

docker file 语法

参考:

Dockerfile reference: https://docs.docker.com/engine/reference/builder/

一个官方提供的docker file的说明，提供了例子和参考以及使用的情况等说明。

FROM

指的是所需要建立的image是基于那个base image来构建的。有时候我们需要从头开始制作 image 就可以使用 scratch ，但是更多的时候，我们是希望基于某个官方制作的image上构建我们自己的image，这时候就应该用 from centos 或后面带上标签来制定版本。尽量使用官方的image作为base image。

LABEL

设置一些描述性信息，metadata不可少，用来描述image的使用情况，有点类似于代码中的注释。

RUN

为了美观，使用反斜线换行，每一条run的语法都会形成一个新的层，所以为了避免无用的分层，合并多条命令成一行。

WORKDIR

有点类似于 Linux 下的 cd ，使用 workdir 不要使用 run cd ， 尽量使用绝对目录。

ADD and COPY

大部分情况下，COPY 优先于 ADD，ADD 除了 COPY 还有额外的功能（解压），添加远程文件、目录使用 curl 或者 wget 。

ENV

设置常量，并且可以通过 $ 来引用，推荐尽量使用，方便维护。

VOLUME and EXPOSE

用户存储和网络。

CMD and ENTRYPOINT

比较

  • RUN：执行命令并创建新的Image Layer

  • CMD：设置容器启动后默认执行的命令和参数

  • ENTRYPOINT：设置容器启动时运行的命令

CMD

  • 容器启动时默认执行的命令

  • 如果 docker run 指定了其他命令，CMD 命令被忽略

  • 如果定义了多个CMD，只有最后一个会执行

如果正常情况下使用 docker run image ，就会执行 CMD 命令，如果在后面添加命令，如 docker run -it image /bin/bash ， Dockerfile 中的 CMD 命令就不会执行。

ENTRYPOINT

  • 让容器以应用程序或者服务的形式运行
  • 不会被忽略，一定会执行
  • 最佳事件，写一个shell脚本作为ENTRYPOINT

拷贝 sh 文件进入容器中，执行这个 sh 脚本。

  • 编写 docker file 文件

cmd 命令不写，并且 entrypoint 命令只写前面一部分，这样，在执行的时候，执行到这个如果后面不加参数的话就会报错，相反的，就可以在 run 的时候添加自己的命令。

在 run 的时候 --vm 1 --verbose 这些参数被 cmd 接收，并且 entrypoint 指令表示一定会执行的指令，而接收的指令在 cmd 中，这是一个十分流行的编写可执行程序并且接收参数的方式。

container 的一些操作

  • docker exec

对运行中的容器进行一些操作，执行一些命令，或者进行交互式操作，进入shell等。

ip a ：显示正在运行中的容器的IP地址。

  • docker stop

停止运行中的容器。

  • docker rm

这个会删除所有的容器。

-a ： 表示所有； -q ： 表示只显示id

并且如果在启动的时候有设置容器的名字 docker run -d --name=demo lim/flask-hello-world ，也可以通过名字进行删除 docker stop demo ，如果启动的时候没有指定名字，也会随机默认生成一个新的名称。

  • docker start

开启容器，后面接id或名称 docker start demo

  • docker inspect

显示容器的详细信息 docker inspect 101e86916d0f

  • docker logs

显示日志 docker logs demo ，这里的日志表示容器运行时候产生的一些输出。
