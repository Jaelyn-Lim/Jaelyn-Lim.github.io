Python 中的 GIL

GIL：全局解释锁，（gil global interpreter lock （cpython））

python中一个线程对应于c语言中的一个线程。

GIL 使得同一个时刻只有一个线程在一个cpu上执行字节码, 无法将多个线程映射到多个cpu上执行。

这个时候，当我们直接去执行的时候，Python 其实是在编译字节码，如果有这个锁存在的话，在某种程度上，能保证线程的安全。

也有例外，GIL 并不会一直占有，还是会去释放的。

可以发现在多次运行的情况下，total 变量的值有很大的波动。

这个时候就能解释，如果 GIL 一直占有的话，就会一直去执行其中的一个函数1000000，但并没有这样，而是在执行该线程一段时间的时候，就会释放 GIL ，给别的线程使用，具体的原理就是 GIL 在执行的一定数量行数的字节码文件的时候，就会去释放锁资源，给别的线程使用。（或者说以时间片来划分，这个是Python内部的实现问题）。

GIL 会根据执行的字节码行数以及时间片释放 GIL ， GIL 在遇到 IO 的操作时候主动释放。（也是因为这样，所以Python在进行多 IO 操作的时候，也是很适合的）。

与简单的使用全局变量的方式对比，Queue 最重要的一点就是它是线程安全的，内部使用的是 Python 的双端队列 deque ，而不是直接使用锁，而 deque 在字节码的时候就达到了线程安全，在速度上也能得到保证。而在 Queue 来说，get 方法和 put 方法是可以阻塞队列的，对于多线程操作来说也是很实用的。

task_done() 方法和 join() 方法：

task_done()

意味着之前入队的一个任务已经完成。由队列的消费者线程调用。每一个get()调用得到一个任务，接下来的task_done()调用告诉队列该任务已经处理完毕。

如果当前一个join()正在阻塞，它将在队列中的所有任务都处理完时恢复执行（即每一个由put()调用入队的任务都有一个对应的task_done()调用）。

join()

阻塞调用线程，直到队列中的所有任务被处理掉。

只要有数据被加入队列，未完成的任务数就会增加。当消费者线程调用task_done()（意味着有消费者取得任务并完成任务），未完成的任务数就会减少。当未完成的任务数降到0，join()解除阻塞。

其中，注意 task_done() 和 join() 的方法

Semaphore

Semaphore 是用于控制进入数量的锁。

文件， 读、写， 写一般只是用于一个线程写，读可以允许有多个。

举个例子，在做爬虫的时候，有时候，在爬取信息的时候，并不能大量的并发去爬取，会有被封的危险，此时，就要去限制并发时候的线程数量。

功能

可以使用 done() 方法来判断线程是否执行完成，通过 result() 方法获取线程执行的返回值。

还有一个常用的方法 cancel() ，用来取消线程，返回取消成功与否，不过对于执行中和执行完成的线程是不能取消的，只能用于在等待中的线程进行取消操作。

不过有时候我们需要获取已经成功的线程的返回，这个时候会用到 as_completed 这个方法。

当异步线程执行完成之后，主线程就能获得已经执行完成的 task 。在 as_completed 方法内部，也是在不断的循环，判断线程是否执行完成，之后再 yield 返回结果的。

还有一个更简单的方法 map() ，和我们所熟悉的 map() 类似，也是将列表中的数据放到线程池中去执行，最后返回一个结果。

从 map() 源码也可以看到，yield fs.pop().result() 返回的是执行完的结果。

与 as_completed() 函数不同，这个函数所返回的数据是和参数 list 的顺序是一致的，而 as_completed() 函数是哪个线程先完成就返回。

还有一个比较常用的函数 wait() ，能够使主线程进行阻塞。能够将某些或者特定的 task 完成之后才继续主线程的运行。还可以接收一个参数 return_when，表示达到某个条件的时候就停止等待，还有一个参数 time_out， 表示最多等待多少时间。

multiprocessing

进程编程与线程编程有很大的不同。

从实际的运行结果来看，会发现两个不同的输出。在创建进程的时候，会将原来主进程的代码全部拷贝一遍，这两个进程之间是完全隔离的，而在线程中，能够通过全局变量进行通信，这个在进程中是做不到的。并且代码也会接着运行，会在 fork 之后继续运行代码。也就是后面的代码是两个进程都在运行着的。

ProcessPoolExecutor 和 multiprocessing 都可以实现多进程的编程，但是使用后面的方式会比较灵活，也比较底层。前面的进程池的方法和线程池的接口调用很相似，并且底层所使用的也是 multiprocessing 来完成的，不过更加推荐前面的方式实现多线程，设计会更加精良，并且也比较容易。

进程间通信

Queue

这个时候需要注意，我们不能再使用 from queue import Queue 这里面的，要去使用 multiprocessing 这个包里面的队列，而这两个 Queue 其实是很相似的。

  • 共享全局变量不能适用于多进程编程，可以适用于多线程。
  • multiprocessing中的queue不能用于pool进程池。
  • pool中的进程间通信需要使用manager中的queue。

要注意这三个 Queue 的区别。

  1. queue 中的 Queue 可以用于多线程中的通信
  2. multiprocessing 中的 Queue 只可以用于简单的多进程的通信，对于进程池的通信不可用。
  3. multiprocessing 中的 Manager ，在实例化之后有个 Queue ，这个可以用于进程池之间的通信。

Pipe

通过pipe实现进程间通信。

pipe的性能高于queue。因为在使用的时候，Queue 会进行比较多的锁的判断，而 pipe 没有。

这里需要注意：pipe只能适用于两个进程！

如果在两个进程间使用通信的时候，可以优先考虑 pipe 。
