asyncio 并发编程

  • 包含各种系统实现的模块化事件循环
  • 传输和协议抽象
  • 对TCP、UDP、SSL、字进程，延时调用和其他的具体支持
  • 模仿futures模块但适用于事件循环使用的Future类
  • 基于yield from的协议和任务，可以用正常的编码顺序来编写并发代码
  • 当产生阻塞IO的调用时，可以将这个事件转移到线程池
  • 模仿threading模块中的同步原语

在使用高并发使用的时候，使用的模式：事件循环+回调（驱动生成器）+epoll(IO多路复用)，asyncio 是python 用于解决异步 io 编程的一整套解决方案。

常见的框架：tornado、gevent、twisted（scrapy， django channels）

首先，torando 是实现了web服务的框架，与 django 和 flask 不同，后者没有实现web服务，在正真部署的时候会使用 uwsgi 、 gunicorn+nginx 等。

而使用 tornado 则可以直接部署，正真部署的时候会使用 nginx 和 tornado 。

协程的简单使用

采用的方式依旧是循环和事件回调的方式，首先要有一个loop循环，实现监听回调。之后使用 run_until_complete 来运行异步操作，这个函数有类似于 join 的功能。会将所有异步操作完成才会继续执行。

当我们需要进行长时间操作的时候，应该使用 await 。

在使用异步操作的，协程的时候，不应该出现同步的逻辑，例如使用 time.sleep() ， 这个同步的逻辑会使得线程卡住，这和普通的循环没什么区别了。再者， await 后面接的是 Awaitable 的对象。

可以使用 asyncio.wait() 函数，接收一个可迭代的对象，一次性将多个异步调用操作传递执行。

由这个例子也可看出，不管执行多少内容，都是大约2秒就能完成，并发性很高。

这个 loop 和之前提到的自己写的循环 loop 逻辑也是一样的，每次都会去执行里面的函数，当遇到 time.sleep() 这样的同步操作的时候，也会进行等待，这样，loop 循环就会顺序执行，但是如果使用 await 的时候，（await asyncio.sleep(2)）会立马返回一个 future 对象，不会阻塞，在下次调度的时候就会判断 future 对象是否完成，进而继续执行。

协程模式是单线程的。

嵌套协程

在嵌套调用的时候，时序图如下：

task 和 compute 通过 yield from 建立了一个通道。

  1. 一开始， loop 会将 print_sum 这个协程注册到 loop 里面，并且创建一个 task （loop 为 print_sum 创建一个 task ）。通过 task 驱动协程的运行，首先会进入一个 pending 的状态，之后去执行 print_sum 。
  2. 而在 print_sum 中，转向去执行 compute 的操作，此时，print_sum 就会进入到一个 suspended （暂停）的状态，去执行 compute 的操作。
  3. 在输出之后，会进入到另外一个 asyncio 的实例，此时 compute 就会同样进入 suspended （暂停）的状态。
  4. 这个时候，会直接返回到 task ，没有返回给 print_sum ，因为 await 和 yield from 类似，所以 task 就和 compute 建立了一个通道，就直接将暂停的状态信息返回给 task ，task 就会返回给 loop。
  5. 在暂停一秒之后，task 就越过 print_sum ，直接将 compute 唤醒，执行 return 的代码。
  6. 此时，compute 就是 done 的状态了，并且抛出一个 StopIteration 给 print_sum。
  7. print_sum 接收到一个 StopIteration ，就解析里面的返回值，执行 print 。
  8. print_sum 执行完成之后，也会变成 done 的状态，同时，也会抛出 StopIteration 给 task ，task 接收到也会将状态改为 done ，同理 loop 。

future 和 task

这里的 future 是 asyncio 中的 future ，不过其中的原理和线程池（进程池）中的 future 很像。

获取结果，当一个 future 类完成并且获取结果的时候，就会调用这个函数，并且将结果返回到结果队列中。

这个是与线程池不一样的地方，因为只有一个线程的关系，所以只能使用 call_soon 这个方法。

task 是一个 future 的一个子类，作为协程和future之间的一个桥梁。

在 task 初始化的时候会调用一次 _step 。

首先 result = coro.send(None) 向协程 send 一个 None ，来启动协程，而线程不用。所以就抽象出了这个 task 方法，首先就激活协程。

之后，如果捕捉到 StopIteration 异常的话，就会获取协程里面的 return 的值。

所以 asyncio 这个模块为了保持和线程相似的函数调用，就加入了 task 这一层，解决协程和线程不一样的地方，用中间的 task 解决这些问题。
