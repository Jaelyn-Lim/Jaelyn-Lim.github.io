协程和异步IO

并发、并行、同步、异步、阻塞、非阻塞

并发：一个时间段内，有几个程序在同一个CPU上运行，但是任意时刻只有一个程序在CPU上运行。

并行：任意时刻点上，有多个程序同时运行在多个CPU上。（多个程序在多个CPU上同时运行，例如CPU是四核，最大并行数量就是4）

同步：代码调用IO操作的时候，必须等待IO操作完成才返回的调用方式。

异步：代码调用IO操作时，不必等待IO操作完成就能返回的调用方式。

阻塞：调用函数的时候当前线程被挂起。

非阻塞：调用函数的时候当前线程不会被挂起，而是立即放回。

C10K 问题和IO多路复用（select, poll, epoll）

C10K 问题：在1999年提出，如何在一颗1GHz CPU,2G内存，1gbps网络的环境下，让单台服务器同时为1万个客户端提供FTP服务。

  • 上一个10年，著名的C10K并发连接问题

Unix下的I/O模型

阻塞式io会造成大量的时间等待，例如在调用接口的时候，会一直等待数据返回，处于浪费资源的状态，不过编写代码较为简单。

不用等到数据准备好，就能立刻返回，但是如果我们需要在某个条件达到的前提下，就可能需要不断的循环，来确定状态连接的准确性，这个时候还是会在循环中大量消耗CPU资源。如果做计算任务或者再次发起其他的连接请求，这方式的优势就很明显。

监听多个io或句柄状态，当其中一个的状态发生变化的时候，就会返回。

于I/O多路复用(又被称为“事件驱动”)，首先要理解的是，操作系统为你提供了一个功能，当你的某个socket可读或者可写的时候，它可以给你一个通知。这样当配合非阻塞的socket使用时，只有当系统通知我哪个描述符可读了，我才去执行read操作，可以保证每次read都能读到有效数据而不做纯返回-1和EAGAIN的无用功。写操作类似。操作系统的这个功能通过select/poll/epoll/kqueue之类的系统调用函数来使用，这些函数都可以同时监视多个描述符的读写就绪状况，这样，多个描述符的I/O操作都能在一个线程内并发交替地顺序完成，这就叫I/O多路复用，这里的“复用”指的是复用同一个线程。

使用不多，还有最后的异步IO（aio），在性能上不会说比IO多路复用快很多，不过在使用难度上比较多，就目前的来说，应该大部分框架还是使用IO多路复用的比较多。

使用select + 回调 + 事件循环获取http

通过使用 select 的注册和监听，我们可以使用回调和事件循环的机制大大加快http请求的速度。

其中的回调指的是，当某个事件发生的时候，例如 select 的注册事件发生的时候，操作系统是不会帮我们去调用注册的代码的，需要我们写一个事件循环的方式来实现回调，这个回调是我们自己来实现的。

  1. select本身是不支持register模式。
  2. socket状态变化以后的回调是由程序员完成的。

首先来看 selector.select() ，这个 select 和 import select 中的 select.select() 不同，selector 对原始的 select() 方法进行了封装，不用再传递队列进去，其实在注册的时候就已经传递参数了。

在 selector.select() 中，会返回一个列表元组， ready.append((key, events & key.events)) ，一个是 key， 另外一个是事件的与运算，向上找寻 key 的来源，可以发现这个 key 是在注册的时候产生的。

这里可以看到，这个SelectorKey 是用 namedtuple 定义的一个类，其中就有一个属性为 data ，这个属性实际上是将注册时候的回调函数放在这的。

这种方式采用的是 select + 回调 + 事件循环 的方式，并发性很高，采用单线程的方式，因为是非阻塞的方式，当处理好URL之后，进行注册，之后一旦注册监听的事件发生了，就在循环中监听到该事件，执行所注册的回调函数，此时，在下一步和这一部之间就可以同步进行，大大提高性能。

当然，select + 回调 + 事件循环 这种方式也是有很多不足：首先，因为注册回调的特殊性，他需要将代码拆分成函数的形式，使得维护成本增加；异常处理会变得很麻烦，因为注册回调之后，就与原来的方法分开，异常也不能被捕捉；嵌套多层，并且某个环节出错，难以定位；变量维护会很麻烦，例如某个数据需要被多个回调处理；变量共享问题等。

  • 可读性差
  • 共享状态管理困难
  • 异常处理困难

协程

  1. 采用同步的方式去编写异步代码。
  2. 使用单线程去切换任务。
    1. 线程是由操作系统切换的，单线程切换意味着我们需要自己去调度任务。
    2. 不在需要锁，并发性能高，如果单线程内切换函数，性能远高于线程切换，并发性更高。

对比一下，如果使用传统函数调用，过程 A->B->C，假如有一个可以暂停的函数，并且可以在适当的时候恢复该函数的继续执行。那我们就可以保证函数执行顺序的情况下，减少等待时间，大大提高效率。并且在单线程内的切换，效率会比多线程的切换快很多。

这时候就有这么个需求：

  • 我们需要一个可以暂停的函数，并且可以在适当的时候恢复该函数的继续执行。
  • 出现了协程 -> 有多个入口的函数， 可以暂停的函数， 可以暂停的函数(可以向暂停的地方传入值)。

close

在调用 close() 之后还尝试去调用生成器的时候，就会抛出异常：StopIteration

并且还要一点需要注意，如果尝试在生成器函数中捕捉异常的话，在使用 close() 的时候就会异常。

主要机制是，在使用 close 的时候，会在 yield 之后抛出一个异常（GeneratorExit），在下次再次使用 next 的时候，就会异常。当我们去捕获异常的时候，但是并没有处理，不过此时生成器就已经结束了，没有返回东西，就会报错。如果将 2 3 注释，就会正常。

当然，也可以在捕获异常的时候抛出 raise StopIteration 来处理。还有一点需要注意，在捕获异常情况的时候，GeneratorExit 是继承自 BaseException ，一个比 Exception 更加基础的异常，所以如果尝试直接用 BaseException 去捕获的话，就不会抛异常。

throw

这个是和异常相关的函数，抛出的异常其实是第一个 yield 所抛出的异常。这也表明，我们需要在生成器函数中自己处理异常。

这是一个处理简单逻辑的 yield from ，而事实上，除了这简单的情况，也有可能需要处理下面的一些情况，所以 yield from 的完整逻辑应该是：

可能存在的情况：

  1. 子生成器可能只是一个迭代器，并不是一个作为协程的生成器，所以它不支持 .throw() 和 .close() 方法；
  2. 如果子生成器支持 .throw() 和 .close() 方法，但是在子生成器内部，这两个方法都会抛出异常；
  3. 调用方让子生成器自己抛出异常
  4. 当调用方使用 next() 或者 .send(None) 时，都要在子生成器上调用 next() 函数，当调用方使用 .send() 发送非 None 值时，才调用子生成器的 .send() 方法；

总结一下：

  1. 子生成器生产的值，都是直接传给调用方的；调用方通过 .send() 发送的值都是直接传递给子生成器的；如果发送的是 None，会调用子生成器的 __next__() 方法，如果不是 None，会调用子生成器的 .send() 方法；
  2. 子生成器退出的时候，最后的 return EXPR ，会触发一个 StopIteration(EXPR) 异常；
  3. yield from表达式的值，是子生成器终止时，传递给 StopIteration 异常的第一个参数；
  4. 如果调用的时候出现 StopIteration 异常，委托生成器会恢复运行，同时其他的异常会向上 “冒泡”；
  5. 传入委托生成器的异常里，除了 GeneratorExit 之外，其他的所有异常全部传递给子生成器的 .throw() 方法；如果调用 .throw() 的时候出现了 StopIteration 异常，那么就恢复委托生成器的运行，其他的异常全部向上 “冒泡”；
  6. 如果在委托生成器上调用 .close() 或传入 GeneratorExit 异常，会调用子生成器的 .close() 方法，没有的话就不调用。如果在调用 .close() 的时候抛出了异常，那么就向上 “冒泡”，否则的话委托生成器会抛出 GeneratorExit 异常。

所以，推荐使用 yield from 这个方法，这个方法为我们处理了很多异常。

生成器实现协程

生成器可以说是一个可以暂停的函数。

通过 inspect 模块中的 getgeneratorstate(gen) 方法，可以看生成器的状态。一开始没有调用的时候是 CREATED （创建），在调用一次之后就变成了 SUSPENDED （暂停）状态。之后进行最后一次调用之后，状态变成了 GEN_CLOSED 。由此可以知道，通过这一个个的状态机制，生成器实现协程也就变成可能。

对于生成器来说，简单的一个 yield 就实现了两个功能，第一返回值给调用方， 第二调用方通过 send 方式返回值给 gen 。这样来说其实可以看成是一个简单的协程，不仅能够把值传递出去，也能接收外边传递进来的值。

一个简单的伪代码，总的来说，是为了解决之前所存在的问题，主题还是采用循环和事件回调的方式，在此基础上加上协程来实现。
